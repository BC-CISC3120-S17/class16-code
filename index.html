Swing Exercise

Overview

The main objective of this exercise is to allow you to write some 'useful' Swing code.

The Project

You'll be implementing a calculator, but one with a small twist: it calculates in base 4 (quaternary) only. (Wikipedia has way more information than you need.) So the only digits are 0, 1, 2, and 3. And 2 + 2 is not 4, but 10. Among other things, this make some of the GUI set-up less tedious. Another way we'll keep things simple is by using integer division (not least because "decimal digits" in non-decimal bases feel weird). So in base 10, 37 / 4 = 9, right? In quaternary, we'd write that as 211 / 10 = 21.

In the skeleton, I've already set you up with some buttons. I've also given you a skeleton of a class that will keep track of the calculator's "state" and do the actual calculation. Your big task is to think through the event handling and calculational logic. (Note that you can never have a "purely" GUI application—there always has to be some "business logic," too.)

Even though is a fairly simple application, there are several things to think through (and watch out for). Here's a list to get you started:

  • You'll need to add a GUI component that corresponds to the 'screen' of the calculator so that you can 'view' the calculation in progress.
  • You'll need to think about what exactly should happen when each button is pressed. As a start, you can generally think about 'digit' buttons and 'operand' buttons. But perhaps not all digits are created equal--if I hit 0 several times, should the same kind of thing happen as if I hit 1 several times?
  • You'll need to develop Base4CalcState into a class that can remember everything about the current state of the calculation. Basically, any information you need to remember between one button-press and the next should be here. Provide methods that let your GUI both update the state and get elements of the state. It may be tempting to do some "remembering" in the GUI classes: don't! You will make your life easier in the long run if you do all your remembering in one place. (We'll talk about this more in Unit 4.)
  • Once you've decided what each button should do (and what it will need from the state, and what changes it will make to the state), you'll need to write event handlers for each different sort of button behavior.
This may sound like a lot, but once you get thinking/diagramming, you'll see that there aren't so many different things to do. If you realize that some buttons need the same basic behavior, write just one method that takes a parameter.

Feel free to collaborate with teammates on this one. Nothing needs to be handed in for this exercise, but the next project might be based on this in some way.

Of course, you can get this working in any order you like. Maybe you want to get the button-clicks "working" first before you tackle the calculating logic. Maybe you want to get the calculator working in base 10 before you tackle the base 4 thing. Up to you...
